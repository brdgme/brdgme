# cathedral-2 Port Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Port the Go game `brdgme-go/cathedral_1` to a Rust crate `rust/game/cathedral-2` (Track B conversion), register/deploy it, deprecate cathedral-1 (no prior Rust port exists, but the Go GameVersion may be deployed), and update tracking docs.

**Architecture:** Standalone Rust game crate implementing `brdgme_game::Gamer`. Domain: 2-player abstract area-control placement game on a 10x10 board. 29 total pieces (14 for player 0, 15 for player 1 including the neutral Cathedral piece which player 1 must place first). No hidden information - `PubState == PlayerState` (full board + remaining pieces visible to everyone), matching the Go `PlayerState`/`PubState` returning `nil` and both renders reading directly off `Game`.

**Tech Stack:** Rust, brdgme_game, brdgme_markup, brdgme_color, brdgme_cmd, brdgme_fuzz, rand, serde.

## Suspected Go defects (preserve verbatim, document in plan/PR notes - do NOT fix without user approval)

1. **`CanPlayPiece` off-by-one bounds check** (`play_command.go:67`): `if piece < 0 || piece > len(Pieces[player])` should be `>=` - `piece == len(Pieces[player])` passes this guard and then `p := Pieces[player][piece]` (line 73) would panic with an out-of-range index. Unreachable through the normal command parser (`PieceParser` bounds `Int{Min:1,Max:len(Pieces[player])}` then subtracts 1, so valid range is enforced upstream), but reachable if `Play`/`CanPlayPiece` is ever called directly with an out-of-range index (as several ported tests do, calling `g.Command` with in-range input only, so this never fires in the ported test suite either). Port the same (broken) bounds check verbatim as `piece < 0 || piece > pieces_len` (not `>=`); do not special-case it.
2. **Turn does not advance after the Cathedral placement** (`play_command.go:185`): `else if player != 1 || piece != 0 { g.NextPlayer() }` - after player 1 places piece index 0 (the Cathedral), `NextPlayer()` is skipped, so player 1 immediately gets another turn. This reads as intentional (the Cathedral is a "free" opening placement that doesn't cost player 1 a turn), not a bug, but call it out explicitly since it's easy for a porting implementer to "fix" by symmetry with normal turn-passing. Preserve verbatim.
3. **Capture ignores the Cathedral for return-to-hand and size-count purposes, but still flips its tile ownership** (`play_command.go:216-245`): in `CheckCaptures`, the `pieces` map is keyed by `PlayerType{Player,Type}` of every non-owner tile found in the enclosed region; if 0 or 1 distinct `PlayerType` values are present the region is captured. For each captured `PlayerType`, `if pt.Player != PlayerCathedral` triggers `PlayedPieces[pt.Player][pt.Type-1] = false` (returns the piece to the owning player's hand for reuse) and increments `capturedPieceCount`; separately, walking the area tiles, `capturedPieceSize` only increments for tiles where `Player != NoPlayer && Player != PlayerCathedral`. Net effect: a Cathedral tile inside a captured region **is** converted to the capturing player's territory (owner set, `Player` reset to `NoPlayer`) exactly like any other captured tile, but it never counts toward `capturedPieceCount`/`capturedPieceSize` and is never "returned to hand" (there is no hand for a neutral piece). This is very likely intentional (matches physical Cathedral rules: the Cathedral piece cannot be captured/removed, but the territory around/under it still resolves), not a defect - call it out so the Rust log-message counters replicate the same undercounting rather than "fixing" it to include the Cathedral.
4. **Pieces are reusable after capture** (`play_command.go:234`, `PlayedPieces[pt.Player][pt.Type-1] = false`): a captured piece goes back to its owner's available pool and can be replayed later in the game. This is correct real-Cathedral-rules behaviour, not a defect, but is easy to miss when translating `PlayedPieces: HashSet`/`Vec<bool>` semantics - make sure `remaining_piece_size` and `can_play_something` naturally reflect pieces flipping back to unplayed.

If any of these turn out, on user review, to actually be bugs worth fixing, get explicit approval first per the porting correctness rule before changing behaviour.

## Key Go source facts (source of truth: brdgme-go/cathedral_1/, read it directly - this is a summary, not the spec)

**Board & coordinates** (`loc.go`, `board.go`): `Loc{X, Y}`, `X,Y` both `0..9`. `Loc.String()` renders as `'A'+Y` then `X+1` (e.g. `{0,0}` -> `"A1"`, `{5,8}` -> `"I6"`) - row letter A-J (Y), column number 1-10 (X). `ParseLoc` (`play_command.go:271`) is case-insensitive regex `^([a-j])(\d+)$`, row from letter, col from number minus 1, validated via `Loc.Valid()` (`0<=X<=9 && 0<=Y<=9`). `Board` is `map[string]Tile` keyed by `Loc.String()`; `AllLocs`/`LocsByRow` precompute every cell in row-major order. `Dir` is a 4-bit flag enum (`DirUp=1, DirRight=2, DirDown=4, DirLeft=8`); `OrthoDirs` = the four cardinals, `DiagDirs` = the four diagonal combinations (`Up|Right`, `Down|Right`, `Down|Left`, `Up|Left`), `Dirs` = both concatenated (8 directions) used for the outer capture-region walk. `Walk(from, dirs, cb)` is a BFS: visits `from`, then for each neighbour in `dirs` not yet queued/visited and `Valid()`, enqueues it; callback returns `WalkContinue`/`WalkBlocked`/`WalkFinish` to control traversal (blocked prevents expanding further from that node but the node stays visited; finish stops the whole walk).

**Tiles & pieces** (`tile.go`, `piece.go`): `NoPlayer = -1`, `PlayerCathedral = 2` (a third "player" identity used only for the neutral Cathedral piece's tiles). `Tile{PlayerType{Player,Type}, Owner, Text}`. `EmptyTile = Tile{{NoPlayer,0}, NoPlayer, ""}`. A tile with `Player == NoPlayer` is empty (possibly `Owner`-claimed territory); a tile with `Player != NoPlayer` has a piece part sitting on it. `Piece{PlayerType, Positions Locs, Directional bool}` - `Positions` are cell offsets from an origin (the first entry in `Positions` is the origin and gets `Text = strconv.Itoa(Type)` when rendered standalone via `Piece.TileAt`, used only for piece-catalogue rendering, not board rendering). Non-`Directional` pieces (symmetric shapes) still accept a `dir` argument for placement (see `CanPlayPiece`/`Play` - direction is not gated by `Directional`; that field is presently unused outside possibly UI/documentation - confirm during Task 1 whether it's read anywhere at all before deciding whether to port it as dead data).

Piece catalogue (`piece.go:75-358`, `Pieces map[int][]Piece`):
- Player 0: 14 pieces, indices `0..13`, `PlayerType{0, 1..14}`, sizes (positions count) `5,5,5,5,4,4,4,3,3,3,2,2,1,1` (types 1-14).
- Player 1: 15 pieces, indices `0..14`. Index 0 is the **Cathedral**: `PlayerType{PlayerCathedral(2), 1}`, 6 cells. Indices `1..14` are `PlayerType{1, 2..15}`, sizes `5,5,5,5,4,4,4,3,3,3,2,2,1,1` (types 2-15) - i.e. player 1's non-Cathedral pieces are the same 14 shapes/sizes as player 0's, shifted by one Type number and one array index because the Cathedral occupies index 0.
- Rotation: `n=2` for `DirUp`, `n=-1` for `DirRight`, `n=1` for `DirLeft`, `n=0` (default) for `DirDown`. `Locs.Rotate(n)` applies `Loc.Rotate` (a 90-degree rotation, `{X,Y} -> {-Y,X}` per positive step) to every position `n` times (negative `n` rotates the other way).

**Placement rules** (`play_command.go`):
- `CanPlay(player)`: normally `CurrentPlayer == player`; once `NoOpenTiles` is true (see below), both players may play simultaneously whenever they individually have a legal move (`CanPlaySomething(player, LocFilterPlayable)`).
- `CanPlayPiece(player, piece, loc, dir)`: bounds-check piece index (see defect #1 above - preserve as-is); reject if already played; **player 1 must play piece 0 (the Cathedral) before any other piece** (`cathedral piece must be played before any others`); rotate `Positions` per `dir`, translate by `loc`; every resulting cell must be `Valid()` (else "playing there would go off the board"), unoccupied (`Player == NoPlayer`, else "there is already a piece there"), and either unclaimed or claimed by the same player (`Owner == NoPlayer || Owner == player`, else "the other player owns that area").
- `Play`: re-validates via `CanPlay`+`CanPlayPiece`, writes each rotated+translated cell's `Player`/`Type` onto the board, marks the piece played, logs (public) `"{player} played {type} (size {n}) {dir} from {loc}"` with `type`/`size`/`dir`/`loc` bolded (dir word from `OrthoDirNames`, e.g. `down`/`up`/`left`/`right`). If the piece just played is not the Cathedral itself **and** player 1's Cathedral has already been played (`PlayedPieces[1][0]`), run `CheckCaptures(loc)` and append its logs. Then: if no player has any playable piece left (`CanPlaySomething(_, LocFilterPlayable)` false for both), the game ends (`Finished = true`) and a public log summarizes `"The game is finished, remaining piece size is as follows:"` followed by one bolded `"{player} - {n}"` line per player. Otherwise, while not yet in simultaneous mode, check whether any *open* tile remains (`LocFilterOpen`: `Player == NoPlayer && Owner == NoPlayer`, i.e. truly untouched, not merely someone's playable territory) for either player; if none remain, flip to `NoOpenTiles = true` (simultaneous end-phase) with log `"No open tiles remain, players will play the rest of their pieces simultaneously."`; else, unless the play just made was player 1's Cathedral (`player != 1 || piece != 0`), call `NextPlayer()` (see defect #2 - Cathedral placement does not cost player 1 a turn).
- `NextPlayer()`: switches `CurrentPlayer` to the opponent **only if** the opponent currently has a playable piece (`CanPlaySomething(opponent, LocFilterPlayable)`); otherwise the current player keeps going (so one player can take several turns in a row if the other is stuck, prior to simultaneous mode).
- `WhoseTurn()`: while not `NoOpenTiles`, returns `[CurrentPlayer]`; once `NoOpenTiles`, returns every player index for which `CanPlaySomething(p, LocFilterPlayable)` is true (0, 1, or both - simultaneous turn resolution, mirroring the seven_wonders-style multi-active-player pattern called out in GAME_PORTING.md gotchas).

**Capture logic** (`CheckCaptures`, `play_command.go:193-267`): triggered after a non-Cathedral placement once the Cathedral has been played. `player := Board[loc].Player` (the mover's colour). BFS-walk (`Walk(loc, OrthoDirs, ...)`) outward through the mover's own contiguous piece-tiles: cells already visited or already `Owner == player` are blocked (already-claimed, no need to re-walk); cells that are an extension of the player's own pieces (`Player == player`) continue the walk; any other cell (empty-unclaimed, opponent-claimed, or opponent-piece) triggers an **inner** BFS (`Walk(l, Dirs, ...)` - all 8 directions this time) that floods the connected non-mover region (blocked by visited cells or by the mover's own piece-tiles, which act as walls) collecting `area` (every cell in the region) and `pieces` (the distinct `PlayerType` set of every non-empty tile found in that region). If the region touches **0 or 1** distinct piece identities (`len(pieces) <= 1`) it is captured: every non-Cathedral piece type found gets returned to its owner's hand (`PlayedPieces[owner][type-1] = false`) and counted (`capturedPieceCount`, `capturedPieceSize` - see defect #3 for the Cathedral carve-out), then every cell in `area` becomes `Owner = player` empty territory (`Board[cell] = EmptyTile with Owner=player`) regardless of whether it previously held the Cathedral, an opponent piece, or was simply unclaimed empty space. If a region has 2+ distinct piece identities it is **not** captured and is left untouched. Public log if any capture happened: `"{player} captured an area of {n}"` plus, only if at least one piece was returned, `" and returned {count} pieces with a combined size of {size}"`.

**Scoring / end of game / status** (`game.go`): `RemainingPieceSize(player)` sums `len(Positions)` over every piece of that player not yet in `PlayedPieces`. `Points()` returns `[]float32{RemainingPieceSize(p)}` per player (raw remaining area, not negated - this is the "current score" display, distinct from the placings metric). `IsFinished()` = `Finished` flag (set inside `Play`, see above). `Placings()`: `metrics[p] = []int{-RemainingPieceSize(p)}`, fed to `GenPlacings` which ranks **descending** by metric (see Global Constraints) - so the smallest remaining area wins (place 1), matching normal Cathedral scoring (fewest unplaced tiles wins). `Status()`: `Finished{Placings}` if `IsFinished()`, else `Active{WhoseTurn}`.

**Command grammar** (`command.go`/`play_command.go`): single command `play <piece> <loc> [<dir>]`.
- `piece`: 1-based integer, `Int{Min:1, Max:len(Pieces[player])}`, mapped to 0-based index by subtracting 1.
- `loc`: `Enum` over every `AllLocs[i].String()` (100 values, `"A1".."J10"`).
- `dir` (optional, `Opt` + `AfterSpace`): `Enum` over `OrthoDirNames` (`up`,`right`,`down`,`left`); defaults to `DirDown` if omitted.
- `CommandParser(player)` returns `nil` (no commands available) unless `CanPlay(player)`; when available it's a single-branch `OneOf` wrapping the `play` parser (so `Command(player, input, players)` errors `"not expecting any commands at the moment"` if it's not that player's turn/phase). `PlayCommand` action always sets `CanUndo: true` (placement reveals no hidden information and is always safely reversible), and passes through the parser's `Remaining` string per the standard chaining rule.
- Parser gating happens once at `CommandParser` construction (`CanPlay`) and the action itself does not re-check `CanPlay` before mutating (unlike the double-check pattern GAME_PORTING.md recommends) - `Play()` does call `g.CanPlay(player)` again as its first line (`play_command.go:108`), so the defence-in-depth check is present, just structured slightly differently (checked inside the shared `Play` method rather than inside a `PlayCommand`-specific wrapper). Port both checks (parser-availability + inside `play`) as usual.

**Render layout** (`render.go`, `board.go`): a full-board ASCII/box-drawing rendering plus a "remaining tiles" catalogue per player. This is the highest-risk area for the parity trap - **read `render.go` and `board.go` directly in Task 2**, this summary only orients:
- `Board.Render()` (`board.go:65-106`) builds a `render.Table(cells, 0, 0)` (**zero row/col spacing - no manual spacer cells needed here**, unlike most other ports) with: a bold header row of box-drawing top corners/edges (`WallStrs[DirDown|DirRight]`, then per-column a bold run of `TileWidth`(6) horizontal-wall characters, then `WallStrs[DirDown|DirLeft]`); one body row per board row, each wrapped left/right by `SideWall` (a 3-line vertical-wall column, `TileHeight`=3 rows tall) and, per cell, either `RenderTile` (occupied) or `RenderEmptyTile` (empty, shows the loc string in owner colour or grey if unclaimed); a mirrored bold footer row of bottom corners/edges. Cells are multi-line strings (each tile render is `TileWidth`(6) x `TileHeight`(3) characters) - `render.Table` here is being used purely to align a fixed grid of equal-sized multi-line blocks, not to pad variable-width text columns, so the "spacer cell" trap from RENDER_PARITY.md is **not** the primary risk here (colSpacing=0), but the multi-line-cell alignment itself (does `Node::Table` handle multi-line cell contents identically to Go's `render.Table`?) must be verified directly against the Go CLI output.
- `RenderPlayerTile`/`RenderCorner` compute box-drawing walls per-cell based on `OpenSides` (whether the neighbour in each of the 8 directions is the *same* `PlayerType` - open sides render as blank piece-background instead of a wall, visually merging same-piece tiles into one shape) - corner rendering is a small local algorithm (`RenderCorner`) that decides `+`/`-`/`|` based on which of the two contributing wall-directions are open; port this exactly, it is easy to get subtly wrong.
- `RenderEmptyTile` centers the loc string vertically or shows it grey (unclaimed) / player-coloured-bold (claimed territory).
- `PlayerRender(pNum)` = board render, then a line of instructional text ("All pieces are shown in their {{b}}down{{/b}} position and pivot around the number."), then a bold `"{player} remaining tiles:"` header + `RenderPlayerRemainingTiles(pNum)`, then the same for the opponent (`Opponent(pNum)`) - **both players' remaining-tile catalogues are shown on every render, including the pub render and each player's own render** (there is no hidden information to redact). `PubRender()` is literally `PlayerRender(0)` - i.e. the "public" render is player 0's render verbatim (same remaining-tiles ordering, own-then-opponent), not a neutral/symmetric view; confirm the Rust `PubState`/`PlayerState` split preserves this (`PubState` should render identically to `PlayerState(0)`) rather than inventing a neutral layout.
- `RenderPlayerRemainingTiles(pNum)`: for every unplayed piece of that player, in index order, render the piece shape (`Piece.Render()`, its own zero-spacing `render.Table`) and pack them into rows via `render.Table(cells, 0, 2)` (**colSpacing=2 - this one DOES need manual spacer cells in Rust**), wrapping to a new row whenever the running width would exceed 10 columns (`curWidth+pWidth > 10`); if the player has no unplayed pieces at all, render grey `"None"` instead of an empty table.
