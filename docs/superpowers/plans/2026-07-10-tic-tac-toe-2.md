# Tic-tac-toe 2 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use
> `superpowers:subagent-driven-development` to implement this plan task-by-task.
> Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Port the old Go tic-tac-toe game to a deployed Rust `tic-tac-toe-2`
game service while correcting its starter/winner attribution bug.

**Architecture:** A standalone `Gamer` crate stores a 3x3 typed board, the
randomly selected starting player, the current player, and a serialized seeded
`GameRng`. One `play <a-i>` parser drives deterministic moves. Public and player
states share the same markup renderer because the game has no hidden state.

**Tech Stack:** Rust 2024, `brdgme_game`, `brdgme_cmd`, `brdgme_markup`, serde,
Kubernetes/Kustomize, Docker Bake.

## Global Constraints

- Work directly on `master`; leave all changes uncommitted and do not push.
- The crate, binary, image, service, and GameVersion are named `tic-tac-toe-2`.
- No `tic-tac-toe-1` service or GameVersion exists in this repository, so no
  deprecation manifest is added.
- Preserve the Go command syntax exactly: `play <a-i>`, case-insensitive.
- Preserve the rendered layout and markup semantics: blue empty-square letters,
  bold lowercase `x`/`o`, gray `|` separators, and three newline-separated rows.
- The random starter is `x`; the other player is `o`.
- Correct the Go winner bug: an `x` line belongs to the selected starter even
  when the starter is player 1. Cover this with a regression test.
- Draws produce placings `[1, 1]` and points `[0.0, 0.0]`; a winner receives
  placing 1 and 1 point while the loser receives placing 2 and 0 points.
- Successful moves emit no logs, are undoable, preserve parser remaining input,
  and advance the current player even when the move finishes the game.
- Port all ten Go tests 1:1 by name and add focused Rust API, status, render,
  command, serialization, and regression coverage.

- [x] **Step 3: Implement the minimal game model and command parser**

Use these signatures and state relationships:

```rust
pub const BOARD_SIZE: usize = 3;
pub const NUM_PLAYERS: usize = 2;
pub type Board = [[Cell; BOARD_SIZE]; BOARD_SIZE];

pub struct Game {
    pub players: usize,
    pub current_player: usize,
    pub start_player: usize,
    pub board: Board,
    pub rng: GameRng,
}

impl Gamer for Game {
    fn start(players: usize, seed: u64) -> Result<(Self, Vec<Log>), GameError>;
    fn command(&mut self, player: usize, input: &str, players: &[String])
        -> Result<CommandResponse, GameError>;
}
```

`Loc` displays one lowercase letter from `a` through `i`; `all_locations()`
returns row-major locations. Parse `play` with `Token`, `AfterSpace`,
`Enum::exact`, `Chain2`, `Map`, and `Doc`. Keep parser availability and action
validation as separate guards. Seed and store `GameRng`, then select the starter
with `random_range(0..NUM_PLAYERS)`.

- [x] **Step 4: Implement status, points, states, and exact shared renderer**

`winner()` maps `Cell::X` to `start_player` and `Cell::O` to the other player.
Active status exposes only `current_player`; finished status uses `gen_placings`
over winner metrics or two equal zero metrics for a draw. `PubState` contains
all visible fields; `PlayerState` wraps it with the requesting player index.
Both renderers delegate to one node builder producing the old three-row board.
