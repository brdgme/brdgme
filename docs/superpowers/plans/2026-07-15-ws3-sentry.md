Date: 2026-07-15
Spec: docs/superpowers/specs/2026-07-15-wasm-prod-errors-design.md (WS3

## Verified facts (research pass, 2026-07-15)

- **Rust `sentry` crate 0.48.5** (latest). Default features:
  `["backtrace", "contexts", "debug-images", "panic", "transport",
  "release-health"]`; default transport is `reqwest` + `native-tls` - NOT
  rustls. Sentry's own code has zero rustls/CryptoProvider touch points; all
  TLS is delegated to reqwest's feature-driven config. **Conclusion: stay on
  the default native-tls transport, skip the rustls feature entirely** -
  avoids the whole crypto-provider class of bug from CODING.md's rustls
  section, no `install_default()` needed for sentry's own sake. `web`'s
  existing `reqwest` dep already pulls `rustls` (aws-lc-rs) for its own use;
  that's unrelated and already immune per CODING.md.
- `sentry-tower` 0.48.5 (feature `http` for the axum-compatible tower
  `Layer`, pulled in automatically via sentry's `tower-http` feature).
  `sentry-tracing` 0.48.5 (tracing_subscriber-compatible `Layer`).
- `@sentry/browser` 10.65.0 + `@sentry/wasm` 10.65.0 (npm latest, matching
  versions). `wasmIntegration()` is exported from `@sentry/wasm`, used in
  `Sentry.init()`'s `integrations` array. Standard npm packages, no
  CDN-only bundle - bundleable via esbuild.
- `wasm-split`: prebuilt binary from **getsentry/symbolicator** GitHub
  releases (NOT a cargo tool; a same-named but unrelated crate exists on
  crates.io from GoogleChromeLabs - do not use that one). Latest
  symbolicator release **26.6.0**. Linux x86_64 asset:
  `https://github.com/getsentry/symbolicator/releases/download/26.6.0/wasm-split-Linux-x86_64`
  - dynamically linked glibc ELF (needs libc/libgcc_s, present on
  `debian:bookworm-slim`; verify `libgcc-s1` is installed in the stage that
  runs it).
- `sentry-cli` latest **3.6.0**. Docs confirm Organization Auth Tokens are
  required for `debug-files upload`. Exact minimal scope for
  `debug-files upload` + `releases new` is not spelled out as a discrete
  list in current Sentry docs; the closest documented match is the
  consolidated **`org:ci`** scope (`docs.sentry.io/api/permissions/`:
  "Use the `org:ci` scope for CI and deployment workflows, including
  uploading source maps, creating releases, and managing code mappings").
  Recommend Michael create an **Organization Auth Token scoped to `org:ci`**
  as `SENTRY_AUTH_TOKEN`; fall back to `project:releases` + `org:read` if
  `org:ci` alone proves insufficient in practice.
- No git-SHA-to-running-app plumbing exists anywhere today (checked
  Dockerfile, Cargo.toml, k8s). `docker-bake.hcl`'s `SHORT_SHA` variable is
  only used for image tags. New plumbing needed: Docker `ARG SHORT_SHA`
  threaded into the `web-builder`/`web` stages as `ENV SENTRY_RELEASE`,
  read by both the SSR init snippet and the server's `sentry::init`.
- k8s: **`k8s/prod/app/web-patch.yaml`** (in the `brdgme` repo, not
  `brdgme-config`) is the existing, unambiguous pattern for non-secret env
  vars on the web deployment - `OTEL_EXPORTER_OTLP_ENDPOINT` and
  `OTEL_TRACES_SAMPLER_ARG` are wired there today as plain `env:` literals
  in a Kustomize strategic-merge patch (applied via
  `k8s/prod/app/kustomization.yaml`). `brdgme-config` carries **no**
  non-secret env values at all - it only pins a remote-base ref
  (`prod/kustomization.yaml` -> `k8s/prod?ref=<sha>`) plus SealedSecrets and
  image tags. Since the DSNs are non-secret, `SENTRY_DSN_WEB` and
  `SENTRY_DSN_SERVER` go into `web-patch.yaml` as literals, replicating the
  OTEL pattern exactly. Nothing changes in `brdgme-config`. Zero existing
  Sentry references in either repo (grepped).
- Node v24 via devenv (`languages.javascript`), no version pin needed
  beyond that; no esbuild currently anywhere in the repo - new dependency,
  installed via the new package.json's devDependencies and invoked from the
  Dockerfile `web-builder` stage (nodejs/npm already installed there).
- `docs/CODING.md` line 5 is the repo's existing "document the exception"
  contract - the Sentry non-Rust-dependency and hosted-SaaS exceptions
  belong there as a new paragraph. `docs/decisions/` (format modeled on
  `LEPTOS_RISK.md`: trigger -> findings -> decision -> consequences) is
  where a fuller record of the hosted-SaaS choice goes; VISION.md has no
  "exceptions" section and is not the right place (confirmed by survey -
  its closest precedent, Resend, is justified inline under "Planned
  Features > Email", not as a formal exception record).
- `rust/web/tests/ssr_pages.rs`'s `get()`/`assert_clean_html_body()`
  helpers (in-process router via `build_router` + `tower::ServiceExt`) are
  the right mechanism to assert the Sentry snippet's presence/absence in
  SSR HTML without a browser.

## Architecture decisions

- **JS bundle location**: `rust/web/js/` (package.json + `sentry.ts` entry).
  esbuild bundles to an IIFE exposing `window.Sentry` (from `@sentry/browser`)
  and `window.SentryWasmIntegration` (the `wasmIntegration` export from
  `@sentry/wasm`), output to `rust/web/public/sentry.js` so cargo-leptos's
  existing assets-dir sync (`assets-dir = "public"`) ships it to
  `site/sentry.js` with no separate copy step.
- **Two script tags in `shell()`**, both before `<HydrationScripts>`,
  following the `THEME_BOOT_SCRIPT` SSR-conditional pattern:
  1. `<script src="/sentry.js"></script>` - only emitted when
     `SENTRY_DSN_WEB` is set.
  2. `<script inner_html=sentry_init_snippet></script>` - inline
     `window.Sentry.init({dsn: "<real DSN>", integrations:
     [window.SentryWasmIntegration()], sendDefaultPii: false, release:
     "<sha-based string>", beforeSend: <scrub fn>})`. **No
     `tracesSampleRate` key at all** (omitted, not zero). Unset
     `SENTRY_DSN_WEB` -> neither tag is emitted (dev/Tilt unaffected).
- **Release string**: `sha-<SHORT_SHA>` (matches the existing image tag
  scheme `sha-<SHORT_SHA>` used in docker-bake.hcl/ci.yml), threaded via a
  new Docker `ARG SHORT_SHA=dev` (top of Dockerfile, redeclared in
  `web-builder` and `web` stages) -> `ENV SENTRY_RELEASE=sha-${SHORT_SHA}`
  in the final `web` stage, read via `std::env::var("SENTRY_RELEASE")` by
  both `shell()` (SSR-embedded into the inline init snippet) and
  `main.rs`'s `sentry::init` call (`release` field).
- **Server-side**: `sentry` (default features, no rustls), `sentry-tower`
  (`http` feature via `sentry`'s `tower-http` feature), `sentry-tracing`.
  `sentry::init()` guard held in a local binding in `main` for the process
  lifetime (same shape as the existing `_tracer_provider` binding).
  `sentry_tracing::layer()` added to the existing `tracing_subscriber`
  registry alongside `fmt_layer`/`otel_layer`. `sentry_tower`'s
  `NewSentryLayer`/`SentryHttpLayer` added to the axum router in
  `router.rs`, same layering position as the existing `TraceLayer`. DSN via
  `SENTRY_DSN_SERVER`; unset -> `sentry::init` still runs but with an empty
  DSN, which the SDK treats as a no-op client (same "disabled when unset"
  convention as the OTLP layer) - confirm this no-op behavior in
  verification rather than assuming it; if the crate instead requires
  skipping `init()` entirely when unset, gate the call on
  `std::env::var("SENTRY_DSN_SERVER").is_ok()` instead.
  `send_default_pii: false` explicitly set (deviation from the SDK's
  quickstart default of `true` - documented in the final report).
- **DWARF pipeline** (`rust/Dockerfile`, `web-builder` stage): add
  `--wasm-debug` to the `cargo leptos build --release` invocation so
  wasm-bindgen preserves DWARF (correcting the WS2-era assumption that `-g`
  alone suffices - wasm-bindgen strips DWARF before wasm-opt runs, so
  `--wasm-debug` is required upstream of `-g`). Install `wasm-split` (pinned
  `26.6.0`, URL above) into the `web-builder` stage. After `cargo leptos
  build`, run `wasm-split target/site/pkg/web.wasm -o
  target/site/pkg/web.wasm --strip --debug-out=web.debug` - strips only
  `.debug_*` sections, preserves the name section (verified in wasm-split
  source per spec), injects a build_id for Sentry's matching. New `FROM
  scratch AS web-debug` stage `COPY --from=web-builder
  /app/web/target/site/pkg/web.debug /web.debug` for CI extraction.
- **CI** (`.github/workflows/ci.yml`, `build-rust` job): after the existing
  `docker/bake-action` step, add a second bake invocation targeting the new
  `web-debug` bake target (local output) to extract `web.debug` onto the
  runner, then a step running `sentry-cli debug-files upload` gated on
  `secrets.SENTRY_AUTH_TOKEN != ''` (skip cleanly, no failure, when absent -
  Michael has not created this secret yet). Also `sentry-cli releases new
  sha-<SHORT_SHA>` (or fold into the same upload step) to align the release
  string. New `docker-bake.hcl` target `web-debug`
  (`output = ["type=local,dest=out/web-debug"]`, `target = "web-debug"`,
  same `SHORT_SHA` arg as `web`).
- **Cargo.toml `args`/build-arg plumbing risk**: `docker-bake.hcl`'s image
  target is a matrix over all ~26 targets; passing `SHORT_SHA` as a build
  `args` value that only `web-builder`/`web`/`web-debug` stages declare via
  `ARG` is expected to be harmless for the other stages (unconsumed
  `--build-arg` is not a hard error in buildx), but this must be verified
  in the build-verification phase, not assumed - if it produces build
  warnings/failures for non-web targets, scope the arg to a per-target
  block instead of the shared matrix target.

## Deliverables -> files touched

1. Frontend bundle: `rust/web/js/package.json`, `rust/web/js/sentry.ts` (or

   `rust/web/public/sentry.js` pre-`cargo leptos build` (so it's picked up

   `<HydrationScripts>`, `SENTRY_DSN_WEB`/`SENTRY_RELEASE` env reads
   (SSR-only - needs a `#[cfg(feature = "ssr")]`-safe env read since
   `shell()` compiles for both hydrate and ssr; check how
   `THEME_STYLE_CSS`/existing code handles ssr-only env reads, likely via a
   value computed only under `#[cfg(feature = "ssr")]` and passed in, or a
   client-side no-op branch).
3. `rust/Dockerfile`: `--wasm-debug` flag, `wasm-split` install + invoke,
   `ARG SHORT_SHA`, `ENV SENTRY_RELEASE`, new `web-debug` scratch stage.
4. `docker-bake.hcl`: `SHORT_SHA` arg wiring to the `web`-related stages,
   new `web-debug` target.
5. `.github/workflows/ci.yml`: debug-artifact extraction + gated
   `sentry-cli debug-files upload`/`releases new` steps in `build-rust`.
6. `rust/web/Cargo.toml`: `sentry`, `sentry-tower`, `sentry-tracing`
   (ssr-feature-gated, default features, no rustls).
7. `rust/web/src/main.rs`: `sentry::init` guard, `sentry_tracing::layer()`

10. `k8s/prod/app/web-patch.yaml`: `SENTRY_DSN_WEB`/`SENTRY_DSN_SERVER` env
    literals with the real DSNs Michael provided.
11. `docs/CODING.md`: new exception paragraph (Sentry JS SDKs as the sole
    non-Rust runtime dependency; hosted Sentry SaaS as an operational-tool

12. `docs/decisions/SENTRY_SAAS_EXCEPTION.md` (new, LEPTOS_RISK.md-modeled
    but shorter): FSL SaaS, free Developer tier, $0 spend cap, operational

## Execution order (serial subagents; Dockerfile/CI/.env.template are
shared-file bottlenecks, so sequence around them)

1. **Phase A - frontend bundle + shell() wiring.** New `rust/web/js/`
   package, Dockerfile esbuild step (base insert, before the DWARF changes
   land so Phase B's diff is additive), `app.rs` script tags, `.env.template`
   `SENTRY_DSN_WEB` line, `ARG SHORT_SHA`/`ENV SENTRY_RELEASE` plumbing +
   `docker-bake.hcl` arg wiring for the `web` target only initially.
2. **Phase B - DWARF pipeline (after A, same Dockerfile).**
   `--wasm-debug`, `wasm-split` install/run, `web-debug` scratch stage,
   `docker-bake.hcl` `web-debug` target.
3. **Phase C - CI wiring (after B).** Debug-artifact extraction bake call +
   gated `sentry-cli` upload steps in `ci.yml`; actionlint or careful
   manual review of the new YAML.
4. **Phase D - Rust server (after A, independent of B/C; only shares
   `.env.template`, sequenced after A wrote it).** Cargo.toml deps,
   `main.rs` init + tracing layer, `router.rs` tower layer,
   `SENTRY_DSN_SERVER` line in `.env.template`.
5. **Phase E - k8s + docs (after A-D settle on real env var names/values).**
   `web-patch.yaml`, `docs/CODING.md` exception paragraph,
   `docs/decisions/SENTRY_SAAS_EXCEPTION.md`.
6. **Phase F - verification (after all).** `cargo build`/`clippy`/`test`
   for web ssr + workspace; `docker buildx bake web` (and `web-debug`)
   end-to-end; inspect shipped `web.wasm` (name section present, zero
   `.debug_*` sections, size vs the WS2-recorded `+250KB gzip / +47.4%
   transfer` delta - note: the brief's "5.8MB raw / 778KB gzip" baseline
   figures do not appear anywhere in this repo; use the actual measured
   sizes instead and report them plainly); confirm `web.debug` is non-empty
   and contains DWARF (`file`/`readelf`/`objdump` on it); run the container
   locally, fetch `/` with `SENTRY_DSN_WEB` set vs unset and check the HTML
   for the snippet (or use the `ssr_pages.rs` in-process helper pattern);
   confirm the server boots with `SENTRY_DSN_SERVER` unset (no panic) and,
   if feasible, with it set to a throwaway value (no init error).
