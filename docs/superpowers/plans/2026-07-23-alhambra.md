# Alhambra Deep Audit and Fix (Implementation Plan)

Research + planning doc only. No source was modified. All paths relative to
repo root. Crate is `rust/game/alhambra-1`. Old Go reference at
`~/Development/brdg.me/game/alhambra/`.

The Rust port of Alhambra has critical rendering bugs (coordinate off-by-one,
missing walls, 0-based tile indices), degraded render output (flat text instead
of tile grids), broken autocomplete, and log parity gaps. This spec restores
full parity with the old Go implementation and adds test coverage.

---

## 1. Requirements

- R1: Fix game type name in DB (`alhambra-1` -> `Alhambra`, `seven-wonders-1`
  -> `Seven Wonders`) via one-off migration + yaml update.
- R2: Restore the 1-cell border in `render_grid` so coordinates match what the
  parser expects (players can place tiles where they intend).
- R3: Render walls with box-drawing characters (full parity with old Go).
- R4: Fix tile indices to 1-based in render (matching 1-based command input).
- R5: Render "Tiles to place" and "Reserved" as proper tile grids with walls
  and numbered footers (matching old Go `RenderTiles`).
- R6: Spend command autocomplete uses `Enum::exact` with actual hand cards.
- R7: Fix `suggest.rs` `Spec::Many` arm to handle delimiters (infrastructure
  fix, benefits Splendor and any game using `Many`).
- R8: Log parity - scoring format, starting player message, 2-player Dirk
  message, final placing rich message.
- R9: Test coverage - render snapshots, command parser tests, log content
  assertions, full play-through.

## 2. Current behaviour (old vs new comparison)

### 2a. Coordinate system / border

**Old Go** (`render.go:232-319`): `Grid.Render(border int)` iterates
`min.Y - border ..= max.Y + border + 1` and `min.X - border ..= max.X + border + 1`.
Called with `border=1` from `RenderPlayerGrid` (`render.go:145`). Then
`AddCoordsToGrid` (`render.go:201-230`) wraps the rendered grid with alpha
column headers (A, B, C...) and numeric row labels (1, 2, 3...) on all four
sides. The coordinate parser (`card.rs:544-572` in Rust, `grid_parse_coord`)
maps `a1` to `Vect { x: min.x - 1, y: min.y - 1 }` - i.e. column `a` is the
border cell LEFT of `min.x`, row `1` is the border cell ABOVE `min.y`.

**New Rust** (`render.rs:79-106`): `render_grid` iterates `min.x..=max.x` and
`min.y..=max.y` with NO border. Column headers start at `min.x` (label `a`),
row labels start at 1 for `min.y`. So the parser expects a 1-cell border that
the render doesn't show - coordinates are off by one in both axes.

**Fix**: render `min.x - 1 ..= max.x + 1` and `min.y - 1 ..= max.y + 1`,
adjusting column/row labels to match the parser's coordinate mapping.

### 2b. Walls

**Old Go** (`render.go:232-319`): Each cell in the grid is rendered as a 2x2
character block (upper-left corner pixel + upper pixel + left pixel + centre
pixel). Wall presence between adjacent tiles determines which box-drawing
character to use. `WallStrs` map (`render.go:16-32`) maps direction-bitmask to
characters: `╬ ╣ ╠ ╩ ╦ ╝ ╚ ╗ ╔ ═ ║`. Empty space (no adjacent tile) renders
as `▒` (NoTileStr). The centre pixel shows the tile abbreviation (3 chars).

**New Rust** (`render.rs:79-106`): Renders only tile-type abbreviations in a
simple table. No walls, no box-drawing, no `▒` fill. Walls are CORE to
Alhambra - they determine placement validity and scoring (longest external
wall).

**Fix**: Port the Go `Grid.Render(border)` algorithm to Rust, producing a
character grid with box-drawing walls. The `Tile.walls: HashMap<Dir, bool>`
field already exists in the Rust code (`card.rs:217`).

### 2c. Tile indices

**Old Go** (`render.go:46-53, 150-155`): `RenderTiles` footer uses `i+1`
(1-based). Reserve footer also `i+1`.

**New Rust** (`render.rs:241, 257`): `enumerate()` produces 0-based indices
(`format!("{}:", i)`). But the command parser (`command.rs:162`) maps
`Int::positive()` with `|n| (n - 1) as usize` - so commands are 1-based.
Players see `0: Pav 1: Ser` but must type `place 1 ...` for the first tile.

**Fix**: Use `i + 1` in render (or `enumerate` starting display at 1).

### 2d. Tiles-to-place / Reserve rendering

**Old Go** (`render.go:42-58, 148-175`): Renders tiles as an actual grid
(`Grid.Render(0)`) with walls visible, plus a numbered footer row below each
tile. The footer shows the tile index (1-based) in grey.

**New Rust** (`render.rs:231-261`): Renders as flat inline text:
`"Tiles to place: 0: Pav  1: Ser"`. No walls, no grid structure.

**Fix**: Port `RenderTiles` - build a temporary `Grid` placing each tile at
`Vect { x: i * 2, y: 0 }`, render with border=0 (showing walls), then append
a footer row with 1-based indices.

### 2e. Spend autocomplete

**Current** (`command.rs:125-136`): `Many::some_spaced(CardParser)` where
`CardParser::to_spec()` returns `Spec::Token("card")`. Autocomplete suggests
the literal string "card" - useless.

**Fix**: Build the spend parser from `Enum::exact(hand_card_strings)` where
`hand_card_strings` is the distinct `Card.to_string()` values in the player's
hand (e.g. `["R3", "B5", "Y2"]`). Pattern: Acquire's `play_parser`
(`acquire-1/src/command.rs:84`) uses `Enum::exact(player.tiles.clone())`.

For `Many` (multiple cards), the spec needs `Spec::Many { spec: Enum::exact(...), delim: Space }`.
The suggest fix (A6) makes this work for 2nd+ cards.

### 2f. suggest.rs Many delimiter bug

**Current** (`suggest.rs:96`): `Spec::Many { spec, .. } => suggest_spec(spec, remaining, names)`.
Ignores the delimiter entirely. After typing `"spend R3 "`, the remaining
input passed to the inner spec is `"R3 "` (or the full remaining string),
which doesn't match any card value.

**Fix**: In the `Spec::Many` arm, attempt to consume already-parsed items +
delimiters from `remaining`, then suggest for the last segment. Algorithm:
1. Try to parse items from `remaining` using the inner spec + delimiter.
2. Track how much was consumed.
3. Suggest for whatever remains after the last successfully parsed item +
   delimiter.

This is the test documented as a "known limitation" at `suggest.rs:491-506`.
The fix turns that test into a positive assertion.

### 2g. Log parity

| Log | Old Go | New Rust | Fix |
|-----|--------|----------|-----|
| Starting player | `"%s is the starting player as they got the fewest cards"` (`game.go:166-169`) | Missing entirely | Add after start_player determination in `start_game` |
| 2-player Dirk | `"As you only have 2 players, you will be joined by %s!"` (`game.go:117-120`) | Missing entirely | Add in `start_game` when `players == 2` |
| Scoring header | `"Scoring <abbr>"` per tile type, single log message (`game.go:234-238`) | `format!("{:?} - ", tt)` per score entry, multiple log messages (`lib.rs:256`) | Restructure to single log per round with `"Scoring <abbr>"` headers |
| Wall scoring | `"%s scored %d for their wall"` (`game.go:257-261`) | `"%s scored %d points for walls"` (`lib.rs:276`) | Minor wording: "for their wall" |
| Final place | Single rich message: `"Giving remaining tiles to the player with the most money cards of that currency"` + per-currency lines (`game.go:398-442`) | Per-tile individual logs (`lib.rs:426-438`) | Restructure to single log message |
| Reserve tiles | `"%s added %s to their reserve"` with tile abbrs (`game.go:363-367`) | `"%s moved %d tile(s) to reserve"` (`lib.rs:458-461`) | Use tile abbrs in a comma list |

### A2. Restore border in render_grid + fix tile indices to 1-based

- **Goal**: R2 + R4 - coordinates shown match what the parser accepts; tile
  indices in render match command input.
- **Files**: `rust/game/alhambra-1/src/render.rs`
- **Change**:
  - `render_grid` (`render.rs:79-106`): change iteration bounds from
    `min.x..=max.x` / `min.y..=max.y` to `(min.x - 1)..=(max.x + 1)` /
    `(min.y - 1)..=(max.y + 1)`. This adds the 1-cell border that
    `grid_parse_coord` (`card.rs:544-572`) expects.
  - Column headers: column `a` corresponds to `x = min.x - 1` (the border).
    Current code: `((x - min.x) as u8 + b'a') as char`. New:
    `((x - (min.x - 1)) as u8 + b'a') as char`.
  - Row labels: row `1` corresponds to `y = min.y - 1`. Current code:
    `y - min.y + 1`. New: `y - (min.y - 1)` which equals `y - min.y + 2`.
    Wait - the parser maps row number `n` to `y = n + min.y - 2`
    (`card.rs:555`). So row label for coord `y` is `y - min.y + 2`.
  - `render_place_tiles` (`render.rs:241`): change `format!("{}:", i)` to
    `format!("{}:", i + 1)`.
  - `render_reserve` (`render.rs:257`): same change.
- **Acceptance**: `grid_parse_coord(&grid, "a1")` returns the top-left border
  cell shown in the render. Tile index `1` in render corresponds to
  `place 1 ...` in commands.
- **Tests**: Unit test building a small grid, rendering to plain text,
  asserting that coordinate `a1` maps to the expected position in the output.
  Test that `place 1` refers to the tile shown as `1:` in render.
- **Depends on**: nothing.

### A3. Wall rendering (full parity with Go box-drawing)

- **Goal**: R3 - render walls with box-drawing characters.
- **Files**: `rust/game/alhambra-1/src/render.rs`
- **Change**:
  Replace the simple table-based `render_grid` with a character-grid renderer
  ported from Go's `Grid.Render(border)` (`render.go:232-319`).

  Algorithm (per Go):
  - For each cell `(x, y)` in the extended bounds (with border), produce a
    2-line output (l1 = top row, l2 = bottom row).
  - **Upper-left pixel** (l1, col): check the 4 tiles touching this corner
    (current, up, left, up-left). If any is non-empty, compute a wall bitmask:
    - `DirUp` bit: `dt.has_wall(Right) || ut.has_wall(Left)`
    - `DirDown` bit: `lt.has_wall(Right) || ct.has_wall(Left)`
    - `DirLeft` bit: `dt.has_wall(Down) || lt.has_wall(Up)`
    - `DirRight` bit: `ut.has_wall(Down) || ct.has_wall(Up)`
    - Map bitmask to box-drawing char via `WALL_STRS` table.
    - If no tile touches this corner, render `▒` (NoTileStr).
  - **Upper pixel** (l1, spans tile width): if `ct` or `ut` is non-empty and
    either has a wall between them (`ct.has_wall(Up) || ut.has_wall(Down)`),
    render `═══` (horizontal wall, repeated `TileAbbrLen` times). Else `▒▒▒`
    or spaces.
  - **Left pixel** (l2, col): if `ct` or `lt` is non-empty and either has a
    wall between them (`ct.has_wall(Left) || lt.has_wall(Right)`), render `║`
    (vertical wall). Else `▒` or space.
  - **Centre pixel** (l2, spans tile width): tile abbreviation (3 chars) or
    `▒▒▒` for empty.

  Wall character map (from Go `render.go:16-32`):
  ```
  Up|Down|Left|Right => ╬
  Up|Down|Left       => ╣
  Up|Down|Right      => ╠
  Up|Left|Right      => ╩
  Down|Left|Right    => ╦
  Up|Left            => ╝
  Up|Right           => ╚
  Down|Left          => ╗
  Down|Right         => ╔
  Left|Right         => ═
  Left               => ═
  Right              => ═
  Up|Down            => ║
  Up                 => ║
  Down               => ║
  ```

  The output is a `Vec<String>` (lines) which then gets wrapped with
  coordinate headers (from A2) and converted to `N::text(...)` nodes.

  Use `N::Bold` for wall characters (matching Go's `render.Bold(...)`).
  Use `N::Fg(Grey)` for `▒` fill (matching Go's `render.Gray`).

- **Acceptance**: A grid with walled tiles renders box-drawing characters
  between them. A tile with no walls on a side shows open space (or `▒` fill
  for empty neighbours). Visual output matches old Go for the same grid state.
- **Tests**: Build a grid with known walls, render to plain text, assert
  presence of expected box-drawing characters at expected positions. Snapshot
  test of a multi-tile grid.
- **Depends on**: A2 (border must be in place first so the wall renderer
  iterates the correct bounds).

### A5. Spend command autocomplete (Enum::exact from hand)

- **Goal**: R6 - spend autocomplete suggests actual cards in the player's hand.
- **Files**: `rust/game/alhambra-1/src/command.rs`
- **Change**:
  In `command_parser` (`command.rs:125-136`), replace the spend parser:
  ```rust
  // Before:
  AfterSpace::new(Many::some_spaced(CardParser))

  // After:
  AfterSpace::new(Many::some_spaced(Enum::exact(hand_cards)))
  ```
  Where `hand_cards: Vec<String>` is built from the player's hand:
  ```rust
  let hand_cards: Vec<String> = self.boards[player]
      .cards
      .iter()
      .map(|c| c.to_string())
      .collect::<std::collections::HashSet<_>>()
      .into_iter()
      .collect();
  ```
  The `Enum::exact` parser matches full card strings (e.g. "R3", "B10").
  `Many::some_spaced` allows multiple space-separated cards.

  Note: `Enum<String>` implements `Parser<T = String>`, so the `Map` closure
  changes from `|(_, cards): (String, Vec<Card>)|` to
  `|(_, cards): (String, Vec<String>)|` and we parse each string to `Card`
  inside the map, OR we wrap `Enum::exact(hand_cards)` in a `Map` that
  converts `String -> Card` via `Card::parse`.

  Keep `CardParser` for the `take` command (take uses market cards, not hand
  cards - the market cards are the same for all players and `CardParser` is
  fine there since autocomplete for take is less critical).

- **Acceptance**: `command_spec(player)` for a player with hand `[R3, R4, B5]`
  produces a spec whose suggest for `"spend "` returns `["R3", "R4", "B5"]`.
- **Tests**: Build a game state with known hand, call `command_spec`, assert
  suggestions. Also test that parsing `"spend r3 r4"` still works
  (case-insensitive via Enum).
- **Depends on**: nothing (but benefits from A6 for multi-card suggest).

### A6. suggest.rs Many delimiter fix (infrastructure)

- **Goal**: R7 - `Spec::Many` suggestion handles delimiters so 2nd+ items
  autocomplete correctly.
- **Files**: `rust/lib/game/src/command/suggest.rs`
- **Change**:
  Replace the `Spec::Many` arm (`suggest.rs:96`):
  ```rust
  // Before:
  Spec::Many { spec, .. } => suggest_spec(spec, remaining, names),

  // After:
  Spec::Many { spec, delim, .. } => {
      let mut rem = remaining;
      loop {
          match spec.parse(rem, names) {
              Ok(out) => {
                  let after_item = out.remaining;
                  if let Some(d) = delim {
                      match d.parse(after_item, names) {
                          Ok(d_out) => {
                              rem = d_out.remaining;
                              continue;
                          }
                          Err(_) => {
                              // Delimiter not found - suggest for current position
                              return suggest_spec(spec, after_item, names);
                          }
                      }
                  } else {
                      rem = after_item;
                      continue;
                  }
              }
              Err(_) => {
                  // Can't parse another item - suggest for what's here
                  return suggest_spec(spec, rem, names);
              }
          }
      }
  }
  ```
  This consumes already-typed items + delimiters, then suggests for the
  remaining input at the current position.

  Update the "known limitation" test (`suggest.rs:491-506`) to assert positive
  suggestions:
  ```rust
  #[test]
  fn many_with_delimiter_suggests_after_consumption() {
      let spec = Spec::Many {
          spec: Box::new(Spec::Enum {
              values: vec!["1".into(), "2".into()],
              exact: false,
          }),
          min: None,
          max: None,
          delim: Some(Box::new(Spec::Token(", ".into()))),
      };
      let s = spec.suggest("1, ", &[]);
      assert_eq!(vals(&s), vec!["1", "2"]);
  }
  ```

- **Acceptance**: `Spec::Many` with a space delimiter suggests correctly for
  `"spend R3 "` (suggests remaining hand cards). Existing `Many` tests still
  pass. Splendor and other games using `Many` benefit.
- **Tests**: Update the known-limitation test; add a space-delimited test
  mirroring the Alhambra spend case; ensure existing `many_suggests_from_item_spec`
  still passes.
- **Depends on**: nothing. Independent of A5 but A5 benefits from it.

- **Coordinate system**: The parser (`card.rs:544-572`) maps `a1` to
  `Vect { x: min.x - 1, y: min.y - 1 }`. Column `a` = `min.x - 1`, row `1` =
  `min.y - 1`. The render MUST show this border cell. After A2, column `a`
  in the render corresponds to `x = min.x - 1` and row `1` to `y = min.y - 1`.

- **Wall data structure**: `Tile.walls: HashMap<Dir, bool>` (`card.rs:217`).
  Access via `tile.has_wall(dir)` (`card.rs:241`). The Go code uses
  `map[int]bool` with `DirUp=1, DirRight=2, DirDown=4, DirLeft=8` bitmask
  constants. The Rust code uses an enum `Dir { Up, Right, Down, Left }`.
  The wall-character lookup in A3 needs a bitmask-to-char map; build it from
  combinations of `Dir` values.

- **Markup node types**: `brdgme_markup::Node` variants used: `N::text(s)`,
  `N::Bold(children)`, `N::Fg(color, children)`, `N::Player(idx)`,
  `N::Table(rows)`. For the wall renderer, output is best as a single
  `N::text(multiline_string)` with `N::Bold` wrapping for wall chars.
  Alternatively, build the entire grid as a pre-formatted string and wrap in
  a single `N::text(...)` node (simpler, but loses per-char coloring). The Go
  version colors walls bold and fill grey - to match, use `N::Group` with
  mixed `N::Bold` and `N::Fg(Grey, ...)` nodes per line, or accept a
  simplification where the whole grid is one color. **Decision**: use
  `N::Bold` for wall chars and `N::Fg(Grey)` for `▒` fill, building each line
  as a `Vec<N>` then joining with `N::text("\n")`.

- **Test patterns**: Use `brdgme_markup::plain(&brdgme_markup::transform(nodes, &[]))`
  to get plain-text for assertions (strips color/bold, preserves layout).
  See `cathedral-2/src/lib.rs:550-551` for the pattern.

- **Enum::exact requires ToString + Clone**: `Card` doesn't impl `ToString`
  directly but has `Display`. Use `card.to_string()` to build `Vec<String>`
  for `Enum::exact`. The parser returns `String` values which then need
  `Card::parse` to convert back.

- **Many delimiter in to_spec**: `Many::some_spaced(parser)` produces
  `Spec::Many { spec: parser.to_spec(), min: Some(1), max: None, delim: Some(Space) }`
  in `to_spec()`. The `Space` spec is `Spec::Space`. The suggest fix (A6)
  must handle `Spec::Space` as a delimiter (parse whitespace).

- **Org is `brdgme`** (not `beefsack`) for any image/URL references.

- **Cargo targeting**: Always use `cargo build/check/clippy/test -p alhambra-1`
  (or `-p brdgme-game` for the suggest.rs fix). Never workspace-wide.

- **Clippy gates**:
  - `cargo fmt --all -- --check`
  - `cargo clippy -p alhambra-1 --all-targets -- -D warnings`
  - `cargo clippy -p brdgme-game --all-targets -- -D warnings` (for A6)

- **No DB tests needed** for A2-A8 (pure game logic). A1 needs a migration
  but no `#[sqlx::test]` is strictly required (the migration is a simple
  UPDATE). If adding one, use `scripts/rust-test.sh` for the full suite.
