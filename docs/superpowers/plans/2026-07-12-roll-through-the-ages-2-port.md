# roll-through-the-ages-2 Port Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Port the Go game `brdgme-go/roll_through_the_ages_1` (2,806 non-test lines + 551 test lines - the largest Track B game) to a new Rust crate `rust/game/roll-through-the-ages-2`, register/deploy it, deprecate `roll_through_the_ages-1`, and update tracking docs.

**Architecture:** Standalone Rust game crate implementing `brdgme_game::Gamer`. Domain: 2-4 player dice-and-worker-placement civilization game. Each player has a `PlayerBoard` (city progress, developments, monuments, food, goods, disasters, ships). A single die-rolling/resource-collection turn structure runs through a strict phase sequence (`Preserve -> Roll -> ExtraRoll -> Collect -> Resolve[-> Invade] -> Build -> Trade -> Buy -> Discard -> next player`), with most phases auto-skipping via `Can*` guards when the current player has no legal action, so a single `next`/command call can cascade through several phases and even multiple `NextPhase()` calls' worth of logs in one response. No hidden information at all (`PlayerState`/`PubState` both return `nil` in Go; `PubRender()` is literally `PlayerRender(CurrentPlayer)`).

**Tech Stack:** Rust, brdgme_game, brdgme_markup, brdgme_color, brdgme_cmd, brdgme_fuzz, rand, serde.

## Suspected Go quirks to preserve verbatim (record for user review per the porting correctness rule; do not fix without approval)

1. **`Roll()` bounds check uses `n > l` not `n >= l` in the wrong direction for validity** (`roll_command.go` `Roll`): `if n < 0 || n > l` where `l := len(g.RolledDice)` and dice numbers are meant to be 1-based indices into `RolledDice` - this actually looks correct (1-based, so valid range is `1..=l`, and the check rejects `n < 0` and `n > l`, allowing `n == 0` through uncaught!). Confirm during Task 3: **`n == 0` is never rejected by this validation loop**, yet `n == 0` is not a valid 1-based dice index and `ContainsInt(i+1, diceNum)` would simply never match it (since `i+1 >= 1` always), meaning specifying `0` in the dice list is silently a no-op (matches no die, keeps nothing extra, keeps nothing removed) rather than being rejected outright. However the parser-level `Int{Min: &minI, Max: &maxI}` with `minI := 1` already prevents `0` from ever reaching this validation via the normal command parser, so this is unreachable in practice (matching Cathedral's defect #1 pattern) - preserve the Go validation logic verbatim (do not add a `n == 0` special case) and note this as unreachable-through-the-parser, mirroring how cathedral-2's off-by-one bounds check was handled.
2. **`KeepSkulls` short-circuits `NextPhase()` only when both `len(RolledDice) == 0` AND not (ExtraRoll-phase-with-Leadership)** (`roll_command.go`): this means if the reroll during the extra-roll phase (which can reroll only 1 die per `RollParser`'s `maxU = 1` cap during `PhaseExtraRoll`) produces another skull, `KeepSkulls` removes it into `KeptDice` again, and if `RolledDice` is again empty, this check still fires `NextPhase()` **even during PhaseExtraRoll with Leadership**, which seems to contradict the "player keeps rerolling one at a time as long as they want, or until dice are empty" mental model - re-read closely: the condition to SKIP auto-advancing is `len(RolledDice)==0 && Phase==ExtraRoll && HasLeadership` being FALSE, i.e., it auto-advances whenever dice are empty UNLESS specifically in extra-roll-with-leadership (where the empty-dice case should let the player choose to reroll a kept skull? or is auto-advance actually correct because there's nothing left to reroll?). This needs careful re-reading of `roll_command.go` directly in Task 2 against the two `game_test.go` tests (`TestGame_KeepSkulls_allDisasterSkip`, `TestGame_KeepSkulls_allDisasterLeadership`) which specifically pin this behaviour - port whatever those two tests require verbatim; do not "clean up" the condition based on this summary's paraphrase, which may itself be imprecise. This is the single trickiest piece of control flow in the whole port - budget extra care and extra tests around it in Task 2.
3. **`GainGoods(n)` distributes goods round-robin starting at `GoodWood` every time it's called, not continuing from wherever a previous call left off** (`player_board.go`): each `GainGoods` call (once per turn, from `CollectPhase`, summing all `good`/`skull` dice results into a single `n`) always starts the round-robin at `GoodWood` regardless of prior state - this is presumably intentional (goods dice always fill wood-first each turn) - preserve verbatim, and note the quarrying bonus (`+1 stone` extra) applies only on the FIRST time the round-robin lands on `GoodStone` within a single `GainGoods` call (`quarryingUsed` bool scoped to the call), not once per turn independent of dice count.
4. **`BuyDevelopment`'s goods-total computation double-counts nothing but silently drops duplicate good names in the parsed list** (`buy_command.go`): `usedGoods map[Good]bool` dedupes the `goods []Good` slice before summing `GoodValue`, so `buy leader wood wood` and `buy leader wood` compute the same total (the second "wood" contributes nothing extra) - this matches the natural reading of "select these goods types to spend towards the cost" (spending is by TYPE not by unit, i.e. buying with "wood" spends ALL your wood goods, not one unit), confirm this reading is correct by re-reading `BuyDevelopment`'s final loop (`for _, good := range goods { g.Boards[player].Goods[good] = 0 }` - yes, entire stacks of each named good type are zeroed, confirming "spend this whole good-type stack toward the cost" semantics) and port it exactly, including the dedup-for-display-suffix-only beh986r (the suffix-building loop over `usedGoods` for the log message, separate from the zeroing loop which iterates the original possibly-duplicated `goods` slice - redundant re-zeroing of an already-zero good is harmless but confirms the zeroing loop is NOT deduped, only the totalling and suffix are).
5. **`BuildMonument`'s "first to build" bookkeeping loop has a latent bug if two players complete the same monument in the same amount call** (`build_command.go`): `first := true; for pNum ... { if g.Boards[pNum].MonumentBuiltFirst[monument] { first = false; break } }` runs BEFORE `g.Boards[player].Monuments[monument] += amount` has had a chance to let any OTHER player's build resolve in between (impossible anyway since builds are strictly sequential per player, so this is not actually reachable as a bug - included here only to confirm during Task 3 that the check is a simple "has anyone already got this flag" scan, straightforward to port as-is with a `Vec<bool>`/loop, not a real defect).
6. **`CheckGameEndTriggered`'s "5th development" comment says "5th" but checks `len(Developments) >= 7`** (`game.go`): the in-code comment `// 5th development built` next to `if len(g.Boards[player].Developments) >= 7` is stale/wrong relative to the actual threshold of 7 - this is a comment-only inconsistency with zero behavioural effect (the check itself, `>= 7`, is what must be ported verbatim); flag it in Task 6's port notes as a Go comment inaccuracy, not a logic defect, and do not "fix" the threshold - port `>= 7` exactly. (Double check against the real game rules for player count - standard Roll Through the Ages triggers game end at 6 developments in a 2-3p game or on the last monument in some editions; do not let real-world-rules knowledge override the Go source's `>= 7` - port the source, not the physical game.)

## Key Go source facts (source of truth: brdgme-go/roll_through_the_ages_1/, read it directly - this is a summary, not the spec)

**Phases** (`game.go` `Phase` enum, in strict linear order per `NextPhase()`): `Preserve -> Roll -> ExtraRoll -> Collect -> Resolve -> [Invade only if 4 skulls + Smithing] -> Build -> Trade -> Buy -> Discard -> (NextTurn, back to Preserve for next player)`. Every phase-entry function (`XPhase()`) sets `g.Phase` then immediately checks whether the phase is skippable for the current player (`CanX` false) and if so recurses into `NextPhase()` - so a single top-level command invocation can cascade through many phases and accumulate many phases' worth of logs before returning. `PhaseResolve` is special: it's entered once after `Collect`, resolves food/famine, then disaster dice (skulls: 0-1 none, 2 drought (Irrigation cancels), 3 pestilence (Medicine cancels per-player), 4 invasion (Smithing redirects to ALL opponents via `InvadePhase`, else Wall cancels for self), 5+ revolt (Religion redirects to ALL opponents' goods, else own goods)), THEN calls `NextPhase()` itself (`PhaseResolve` itself is a `case` target for phase-advance dispatch coming FROM both `PhaseCollect` and, after an invade command completes, `PhaseInvade`, per `NextPhase()`'s `case PhaseResolve, PhaseInvade: return g.BuildPhase()` - i.e. once you're past Resolve, the next step is always Build regardless of whether you routed through Invade).

**Dice** (`dice.go`): 6 faces (`Food`=FFF, `Good`=G, `Skull`=GXG worth 2 goods, `Workers`=WWW, `FoodOrWorkers`=FF/WW - player chooses via `take` command, `Coins`=C worth 7 normally or 12 with Coinage). `Roll()`/`RollN(n)` use a package-level `rand.Rand` seeded from wall-clock time in Go - port using this repo's per-game deterministic `rng` field pattern instead (see Global Constraints).

**Roll mechanics** (`roll_command.go`): turn starts with `NewRoll(cities)` rolling `cities` dice (city count starts at 3, +1 per city level crossed: `CityLevels = [3,7,12,18]`, `Cities() = 3 + count(levels <= CityProgress)`). `KeepSkulls()` immediately removes ALL skull dice rolled into `KeptDice` (skulls can never be rerolled, except: "You can reroll skulls in single player" - `PlayerCount()==1` guard, which is dead in this platform since `PlayerCounts()` is `[2,3,4]` and `start` rejects `players<2` - port this guard anyway for source fidelity even though it can never trigger). Player then gets up to `RemainingRolls=2` roll actions (`roll <dice numbers>` - reroll the specified 1-based positions among currently-rolled (non-kept) dice; unspecified positions are kept as-is and merged into `KeptDice` for that pass), each followed by another automatic `KeepSkulls()`. After both rolls (or earlier if `RemainingRolls` hits 0), enters `ExtraRoll` phase: ALL previously-kept dice become rerollable again (`RolledDice = append(RolledDice, KeptDice...)`), but only if the player has the Leadership development (`reroll 1 die`) - `RollParser`'s `maxU` is capped at 1 during this phase. After the (optional) extra roll (or immediately if no Leadership), `CollectPhase` merges everything into `KeptDice` and tallies goods/food/workers/coins, per-die-type, per Global Constraints quirk #3 above.

**Take command** (`take_command.go`): only reachable during `PhaseCollect` when at least one `DiceFoodOrWorkers` die was kept; requires exactly one `TakeFood`/`TakeWorkers` choice per such die (`len(actions) != numDice` errors); food gives `+2 (+1 with Agriculture)`, workers give `+2 (+1 with Masonry)` per action (note: these `+2` bonuses for FoodOrWorkers dice are HALF the `+3` bonus for pure Food/Workers dice results - the ambiguous die is worth less than a dedicated one).

**Build/Trade/Buy/Discard phases** (`build_command.go`, `buy_command.go`, `player_board.go`): Build phase lets the player spend `RemainingWorkers` on city progress (`build <n> city`, capped at `MaxCityProgress=18`) or any in-progress monument (`build <n> <monument>`, capped at that monument's `Size`; completing a monument checks "who built it first" across ALL players via `MonumentBuiltFirst` and triggers `CheckGameEndTriggered` if all 7 monuments are now fully built by someone, or if this player's own 7th distinct development was just bought - note `CheckGameEndTriggered` is called from BOTH `BuildMonument` on monument completion AND `BuyDevelopment` on development purchase), or build ships (1 wood + 1 cloth each, capped at 5 total ships, requires Shipping development) - `CanBuildOrTrade` requires having a legal city/monument OR ship option; if none, auto-advances. Trade phase (`trade_command.go`) is Engineering-only: convert stone to workers 1:3, requires stone > 0 and only usable in the Build phase (NOTE: `CanTrade` checks `g.Phase == PhaseBuild`, not a separate Trade phase gate as the phase name might suggest - re-verify this is not a naming trap: the phase enum has both `PhaseBuild` and `PhaseTrade` as distinct phases, but `TradeCommand`'s guard checks `Phase == PhaseBuild` - meaning trading is available DURING the Build phase, and the separate `PhaseTrade` (Swap via Shipping development) is a DIFFERENT, later phase entirely for swapping goods types via ships, not stone-for-workers trading; do not conflate `TradeCommand`/`TradeStone` (build-phase, stone->workers, Engineering) with `SwapCommand`/`Swap` (trade-phase, goods<->goods, Shipping) - they are two unrelated commands despite similar-sounding names). `PhaseTrade` (`TradePhase()` in game.go) is the ship-based goods-swapping phase (`SwapCommand`, Shipping development, `RemainingShips` budget, 1 ship per unit swapped, requires 2 different good types and available capacity on the target good per `GoodMaximum`). Buy phase (`buy_command.go`) lets the player purchase ONE not-yet-owned development using `RemainingCoins + selected goods' stack values` (goods are selected by TYPE - see quirk #4 - and entire stacks of each selected type are spent/zeroed, not partial amounts); `buy <development> [all|<good>...]`; auto-skips if total possible buying power (`RemainingCoins + GoodsValue()`, plus `Food*6` if Granaries) is under the cheapest development's cost of 10. Discard phase (`discard_command.go`) only triggers if goods exceed the 6-good limit (`GoodsOverLimit`, waived entirely by Caravans development) and forces discarding down to the limit one good-type-and-amount at a time, ending the turn (`NextTurn()`) once at/under the limit.

**Sell/Invade** (`sell_command.go`, `invade_command.go`): Sell (Buy phase only, Granaries development) converts food to coins 1:6. Invade (Invade phase only, reachable solely via the 4-skull disaster path with Smithing) spends spearheads to deal `amount*2` extra disaster points to EVERY other player (Wall blocks per-player).

**Scoring** (`player_board.go` `Score()`): sum of development points (flat per-development, from `DevelopmentValues`) + monument points (`Points` if `MonumentBuiltFirst[m]` else `SubsequentPoints`, only for FULLY built monuments) + bonus points (Commerce: `+1/good` currently held; Architecture: `+2/monument fully built by this player`; Empire: `+1/city`) - `Disasters` (raw subtraction, not scaled). `Points()` (per-turn running score, always computed) = `Score()` directly, not delta/negated. `GoodsValue(good, n) = triangular(n) * (good_index+1)` (wood=1x, stone=2x, pottery=3x, cloth=4x, spearhead=5x multiplier on `n*(n+1)/2`); `GoodMaximum(good) = 8 - good_index` (wood cap 8, stone 7, pottery 6, cloth 5, spearhead 4).

**Command grammar summary** (`command.go`, one `OneOf` gated per-command by its own `Can*`): `next` (advance phase, always available except mid-Collect/Discard/Roll-with-remaining-rolls-left/etc per each phase's specific `CanNext`); `roll <dice#>...` (1-based positions, `Many` bounded by remaining rolled dice, capped at 1 during ExtraRoll); `take <food|workers>...` (one choice per FoodOrWorkers die kept); `preserve` (Preserve phase only, requires Preservation dev + pottery + food>0, doubles food for 1 pottery); `build <n> city|<monument enum>|ship` (three sub-forms combined via nested `OneOf`+`Chain`); `trade <n>` (stone->workers, Build phase, Engineering); `buy <development enum> [all|<good enum>...]`; `sell <n>` (food->coins, Buy phase, Granaries); `swap <n> <good> <good>` (Trade phase, Shipping, ships budget); `discard <n> <good>` (Discard phase, forced); `invade <n>` (Invade phase, Smithing).
