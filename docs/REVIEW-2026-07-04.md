# Comprehensive Project Review - 2026-07-04

Status: COMPLETE.

Scope: vision/architecture alignment, tech choices, application code quality
(rust/web, rust/bot, rust/operator, rust/lib, rust/game, brdgme-go), infra
(k8s, Tiltfile, scripts, devenv), CI, and testing gaps.

Severity legend: [HIGH] fix before cutover / actively risky. [MED] should fix,
not urgent. [LOW] cleanup / nice-to-have. [OK] reviewed, no action.

## 2. rust/web (monolith)

### Duplication

- [HIGH] `game/server.rs` (REST handlers) and `game/server_fns.rs` (Leptos
  server fns) duplicate the same orchestration logic almost line-for-line:
  `create_game`/`create_new_game`, `undo_game`x2, `concede_game`x2,
  `restart_game`x2 - roughly 500 duplicated lines covering game-service call,
  status destructure, DB write, re-fetch, broadcast, bot trigger. Any bug fix
  must be applied twice (the restart double-broadcast logic already exists in
  both). Two clean fixes, pick one:
  1. Extract each flow into `game/mod.rs` (like `execute_command` already
     was) and have both entry points delegate - proven pattern in this repo.
  2. Better: decide whether the REST handlers are needed at all. The Leptos
     frontend uses server fns; the bot uses only
     `/api/internal/game/{id}/command`; legacy clients use `rust/api`, not
     these routes. Verified during this review: no code in `rust/web` (WASM
     or SSR), `web/`, or `rust/bot` calls `/api/game/*`; the bot uses only
     `/api/internal/game/{id}/command`. Delete all of them except the
     internal command route (~600 lines + tests deleted).
- [MED] The `match status { Active {..} => .., Finished {..} => .. }`
  destructure into `(is_finished, whose_turn, eliminated, placings)` appears
  6+ times across server.rs/server_fns.rs/mod.rs. One small helper on
  `brdgme_game::Status` (or in `game/mod.rs`) removes them all.
- [MED] The post-mutation epilogue (find_game_extended -> get_all_game_logs
  -> broadcast -> trigger_bot_turns) is repeated 7 times. Extract
  `broadcast_and_trigger(pool, broadcaster, http_client, game_id, renders)`.
- [MED] `db.rs`: the two giant hand-mapped row->struct blocks in
  `find_game_extended` and `find_active_games_for_user` duplicate the full
  GamePlayer field list. A shared row-mapping helper (or sqlx `FromRow` with
  prefixed aliases) halves the surface where a new column can be missed.
  (Phase 7 added `last_turn_at` etc. precisely because a field was missed.)

### Correctness / reliability

- [MED] `execute_command` (`game/mod.rs:105`): `create_game_logs` runs in a
  separate transaction after `update_game_command_success` commits. A crash
  between the two loses the move's logs permanently (logs are the game
  history for play-by-email). Fold log insertion into the same transaction,
  or accept and document it.
- [MED] Non-atomic restart: `restart_game` creates the new game, then sets
  `restarted_game_id` in a separate statement. A failure in between leaves a
  finished game restartable again -> two successor games. Wrap in one
  transaction (needs the UPDATE inside `create_game_with_users`'s tx or a
  wrapping tx).
- [LOW] `create_game_with_users` reads `find_game_version` via `pool` while
  a `tx` is open (db.rs:646) - works, but inconsistent isolation and one
  extra round trip; the version was already fetched by every caller. Pass
  `game_type_id` in `CreateGameOpts`.
- [LOW] `concede_game` (db.rs) assigns place 1 to every non-conceder; safe
  only because handlers enforce 2-player. A `debug_assert!`/comment in the
  db fn would prevent silent misuse.
- [LOW] `internal_play_command` compares the API key with `!=` -
  non-constant-time. Low risk in-cluster; `subtle::ConstantTimeEq` or length
  guard is a one-liner if it ever fronts the internet.

### Auth

- [HIGH] `confirm_login` (`auth/server.rs:171`) looks up the 6-digit code
  globally: `WHERE login_confirmation = $1`, with no email/user scoping and
  no rate limit on the confirm endpoint (only `login` is limited). An
  attacker can brute-force the 1M code space against ALL pending logins at
  once, and a code collision between two pending users logs one in as the
  other (fetch_optional picks an arbitrary row). Fix: `confirm_login(email,
  token)` scoped to that user, plus rate-limit confirm attempts (the
  limiter infra already exists). Legacy web UI already collects the email
  in step 1, so the Leptos form has it available.
- [OK] Login flow otherwise sound: per-IP governor rate limiting sized
  against the Resend free tier, 1h token expiry, token cleared on use,
  session invalidation via `user_auth_tokens` deletion, secure-cookie env
  switch, SameSite=Lax, 30-day inactivity expiry.
- [LOW] `login` returns success/failure only via `LoginResponse.success` -
  fine - but rate-limit rejection also returns HTTP 200 with a message;
  acceptable for a server fn, just note tower_governor's headers are lost.

### WebSocket

- [MED] `handle_socket` PSUBSCRIBEs every client to `game.*`: every
  connected browser receives every public game update for every game in the
  system. Payloads are the full legacy ShowResponse JSON (KB-scale per
  move). Public-only data so no confidentiality issue, but it is O(clients
  x moves) fan-out that grows quadratically with activity. Phase 17 (NATS)
  should subscribe per-game/per-user only - worth writing into the Phase 17
  design now.
- [MED] No server-initiated WS ping. The receiver half is dropped, nothing
  is ever sent unless a game event fires, so idle connections are at the
  mercy of LB idle timeouts (flagged in Phase 14 prod prereqs as an open
  question). A `tokio::time::interval` ping loop is ~10 lines and makes the
  DO LB question moot. Client-side 2s reconnect exists, but each drop loses
  update continuity.
- [LOW] `websocket.rs` legacy payload builders (~250 lines) are temporary
  Phase 16 compat code - correctly isolated in one file. Tag the legacy
  structs with a `// DELETE at Phase 16 decommission` marker so the cutover
  checklist catches them.
- [LOW] `broadcast_game_update` queries `user_auth_tokens` and per-player
  logs inside the player loop - N+1 per broadcast. Acceptable at current
  scale; goes away with the legacy channels at Phase 16/17.

### Assessment

- [OK] `db.rs` query layer is clean SQLx with compile-checked macros,

- [OK] `execute_command` orchestration is well-factored and thoroughly

- [OK] ELO port matches the stated legacy algorithm; bot-game exclusion and

## 3. rust/bot

- [HIGH] Private log leak into the bot prompt: `load_bot_context`
  (`rust/bot/src/main.rs:389`) fetches `SELECT body FROM game_logs WHERE
  game_id = $1 ... LIMIT 20` with no `is_public`/`game_log_targets` filter.
  The bot's LLM prompt includes private logs targeted at OTHER players
  (hidden hands/info in games that use targeted logs) - an unfair-play bug.
  Reuse the same filter as `db::get_game_logs` (public OR targeted to the
  bot's own game_player id).
- [MED] No HTTP timeouts: `reqwest::Client::new()` has no default timeout.
  A hung LLM provider or game service blocks a bot turn indefinitely (and
  in the monolith, a hung game service pins a request handler). Set
  `.timeout(...)` when building every reqwest Client (web: ~10s for game
  services; bot: a few minutes for the LLM call via per-request timeout).
- [LOW] Dynamic (non-macro) sqlx queries with `try_get(..).unwrap_or(..)`
  defaults silently mask schema drift (e.g. `is_turn` unwrap_or(false)
  turns a query bug into "bot never plays"). Acceptable short-term; the
  Phase 13 rewrite should move to checked queries or explicit errors.
- [LOW] Log messages say "Ollama" but the provider is generic
  OpenAI-compatible (currently OpenRouter) - misleading during debugging.
- [OK] Retry loop, stale-turn double-check, trace_id-threaded structured
  logging, and prompt templating (minijinja, 14 tests) are all sound.
  Phase 13 rewrites the transport; the loop core carries over.

## 7. Infrastructure (k8s, Tiltfile, scripts, devenv)

- [HIGH] No readiness/liveness probes on ANY Deployment (web, bot, all 20
  game services, legacy trio). With `replicas: 2` and rolling updates, the
  Gateway will route to web pods before Leptos is listening -> dropped
  requests on every deploy; a wedged pod is never restarted. Minimum fix
  before prod cutover: readiness+liveness on `web` (a `/healthz` route or
  TCP probe on 3000) and on game services (TCP :80 is enough).
- [MED] No resource requests/limits anywhere. On 3x2GB nodes the scheduler
  is flying blind and one leaky pod can evict the DB. The VISION doc itself
  quotes per-service RSS figures - encode them as requests. Small effort,
  do it with the probe pass.
- [MED] `k8s/base/game/`: 20 directories x 4 near-identical files. Two
  sanctioned fixes: (a) the VISION long-term goal - operator manages
  Deployment/Service from the GameVersion CR - eliminates 3 of 4 files per
  game and is already planned; (b) short-term, a kustomize component with
  per-game name/image patches. Recommend accelerating (a) rather than
  building (b); the operator already owns the CR lifecycle.
- [LOW] Stale `k8s/base/ingress/` and `k8s/base/ingress-nginx/` dirs remain
  (already excluded from kustomizations). Already covered by the Phase 16
  stale-artifact cleanup - fine to leave until then.
- [LOW] `k8s/prod/kustomization.yaml` uses the deprecated `bases:` field;
  swap to `resources:` before kubeconform lands in CI.
- [LOW] Gateway has HTTPS listeners only - no port-80 listener/redirect, so
  `http://brdg.me` will hang rather than redirect. Add an HTTP listener
  with a RequestRedirect filter (cert-manager's HTTP01 solver routes are
  independent of this).
- [OK] Tiltfile is exemplary for its size: three documented modes, dev
  workarounds confined to the Tiltfile per stated convention, tricky bits
  (Cilium NodePort pin, mirrord) explained inline with the *why*.
- [OK] `setup-kind-cluster.sh` post-ctlptl is small, idempotent, and each
  non-obvious flag carries its failure story. Good use of ctlptl over
  hand-rolled bootstrap.
- [OK] devenv.nix pins the whole toolchain including the wasm-bindgen/CLI
  match. `playwright-driver.browsers` is already staged for Phase 11.6.

- [MED] `build-web` and `build-rust-games` hand-roll 5 copies of the

- [LOW] `on: push` + `on: pull_request` double-runs CI for branch PRs. Use

  cancel-in-progress group.
- [LOW] Go games build as one serial 17-image loop in a single job. Fine
  while cached; matrix it only if it becomes the long pole.
- [OK] Service containers for Postgres/Redis, SQLX_OFFLINE compile +
  runtime `#[sqlx::test]` split, gha cache, push-only-on-master gating are
  all correctly set up.
- [OK] Renovate + cargo-deny + kubeconform are already a planned quick win;

## 9. Testing Gaps

Current coverage is genuinely good post-Phase 11: db layer, orchestration,
handler auth, login flow, ELO, contract harness, markup/parser libs. The
remaining gaps, in priority order:

1. [HIGH] E2E/hydration (Phase 11.6, pending): still the only layer that
   catches SSR/hydration panics - the project's documented most dangerous
   failure mode. Nothing new to add; finish 11.6's harness design pass.
2. [MED] `websocket.rs` has zero tests: legacy payload shape (relied on by
   web-legacy during Phase 16 side-by-side!), per-player log filtering (an
   info-leak surface), and channel targeting are all unverified. A
   `#[sqlx::test]` + Redis SUBSCRIBE assertion test is feasible with the
   existing test infra. The legacy-payload half dies at Phase 16, but the
   side-by-side period is exactly when it matters.
3. [MED] Frontend/`app.rs`/`components/game.rs` logic (autocomplete
   filtering, suggestion clicks, effect wiring) is untested and only
   coverable by 11.6 E2E - another reason to prioritize it.
4. [LOW] Bot service: turn loop untested by design (deferred to Phase 13
   rewrite) - agreed, keep deferring.
5. [LOW] Operator: no reconcile tests. A `#[sqlx::test]` against
   `upsert_game_type_and_version` would cover the SQL half without needing
   a kube test env.
6. If the `confirm_login` fix (Section 2) lands, add tests: code scoped to
   wrong email rejected; confirm endpoint rate-limited.

server.rs/server_fns.rs duplication, (b) prod-readiness gaps in manifests
